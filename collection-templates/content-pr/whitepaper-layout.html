Kapitel 1

Einführung

In einer Welt, in der Echtzeitdaten zum entscheidenden Wettbewerbsvorteil werden, stossen traditionelle Request-Response-Architekturen an ihre Grenzen. Event-Driven Architecture (EDA) bietet einen fundamentalen Paradigmenwechsel: Statt synchroner Anfragen reagieren Systeme auf Ereignisse, sobald diese eintreten.

Dieses Whitepaper führt Sie von den theoretischen Grundlagen bis zur praktischen Implementierung einer Event-Driven Architecture mit der {{BRAND_NAME}}-Plattform. Sie erfahren, warum führende Unternehmen auf EDA setzen und wie Sie die gleichen Vorteile für Ihre Organisation nutzen können.

Kernaussage
Unternehmen, die auf Event-Driven Architecture umstellen, berichten von einer durchschnittlichen Reduktion der Verarbeitungslatenz um 73% und einer Steigerung des Datendurchsatzes um das 4-fache.

Warum Event-Driven?

Drei zentrale Treiber machen EDA zur Architektur der Zukunft: die steigende Nachfrage nach Echtzeit-Verarbeitung, die Notwendigkeit loser Kopplung in Microservice-Landschaften und die Anforderung an horizontale Skalierbarkeit bei unvorhersehbaren Lastspitzen.

Kapitel 2

Grundlagen der Event-Driven Architecture

Um Event-Driven Architecture effektiv einzusetzen, müssen wir die grundlegenden Konzepte verstehen. Ein Event ist eine unveränderliche Aufzeichnung eines Geschäftsereignisses – etwas, das in der Vergangenheit passiert ist und nicht rückgängig gemacht werden kann.

Event vs. Command vs. Query

Es ist wichtig, zwischen Events, Commands und Queries zu unterscheiden. Während ein Command eine Aufforderung darstellt, etwas zu tun, und eine Query eine Informationsanfrage ist, beschreibt ein Event etwas, das bereits geschehen ist.

Kapitel 3

Architekturmuster in der Praxis

In diesem Kapitel untersuchen wir die gängigsten Architekturmuster für Event-Driven Systeme und wann welches Muster am besten geeignet ist.

Best Practice
Beginnen Sie mit dem einfachsten Muster, das Ihre Anforderungen erfüllt. Event Notification ist oft der beste Einstiegspunkt, bevor Sie zu komplexeren Mustern wie Event Sourcing übergehen.

Die vier wichtigsten Muster sind: Event Notification, Event-Carried State Transfer, Event Sourcing und CQRS (Command Query Responsibility Segregation). Jedes Muster hat spezifische Stärken und Trade-offs.
